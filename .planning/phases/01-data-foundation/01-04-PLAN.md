|----------|-------------|
| Python → Power BI API (second call) | SKU DAX query — same trust boundary as fee query; same soft-error check required |
| Python → filesystem (output/) | CSV write — output path must not escape the output/ directory |
| Pydantic validation → explore_fees main() | Validation failure must halt execution before CSV write, not after |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | DAX date injection in `build_fee_dax()` | accept | Date parameters are Python `datetime.date` objects converted to integers — no string interpolation of user input; date literals go into DAX as `DATE(YYYY, M, D)` integer args |
| T-04-02 | Denial of Service (data integrity) | CSV written before Pydantic validation | mitigate | `validate_output_df()` called before `to_csv()` in main() step order; if validation raises, CSV is never written |
| T-04-03 | Information Disclosure | CSV output file contains fee data | accept | output/ is gitignored; CSV stays local; no network transmission in Phase 1 |
| T-04-04 | Tampering | run_dax called twice — second call (SKU) also needs soft-error check | mitigate | `run_dax()` always checks `result.get("error")` regardless of which query is passed; the check is in the function, not the caller |
| T-04-SC | Tampering | pip packages | accept | No new packages; pydantic 2.13.4 already verified via slopcheck |
