<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Python process → Power BI REST API | Bearer token in Authorization header; token lifetime ~1 hour |
| filesystem → token cache | `~/.claude/powerbi_token_cache.json` contains refresh tokens for all PBI datasets the user can access |
| .env file → process | Credentials loaded at startup; must not be logged |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | `~/.claude/powerbi_token_cache.json` | mitigate | `get_token()` writes cache with `open(CACHE_PATH, "w")` — no explicit chmod. Add note in code: on Windows, verify Security tab shows only current user. Documented as manual verification in VALIDATION.md |
| T-02-02 | Tampering | `run_dax()` — DAX soft error silently accepted | mitigate | `result.get("error")` check raises RuntimeError before rows are returned; per RESEARCH.md Pattern 2 — this is the primary mitigation for silent partial data acceptance |
| T-02-03 | Denial of Service (data integrity) | `run_dax()` — partial results from row limit violation | mitigate | `validate_value_count()` called before query execution; raises ValueError if projected value count >= 1,000,000 |
| T-02-04 | Information Disclosure | Credentials in logs | mitigate | `get_token()` never logs token values; env var names are logged but not values; `flow["message"]` is safe (contains only device code, not credentials) |
| T-02-SC | Tampering | pip packages | accept | All packages [OK] per slopcheck audit in RESEARCH.md; no new packages introduced in this plan |
</threat_model>
